# PR Review - ADR 0015

## OAuth + Token-File Security Plumbing (ADR 0015 §1)

Scope reviewed:
- `ergodix/oauth.py`
- `ergodix/auth.py`

Focus:
- OAuth flow
- refresh-token persistence
- file-permission handling

---

### Strengths

### Symlink + TOCTOU protection

Both `oauth.py::load_oauth_tokens()` and `auth.py::_read_file_data_checked()` use `O_NOFOLLOW` + `fstat` on the open fd. This is the right pattern — the mode check runs against the same inode being read, closing the TOCTOU gap. Windows fallback (O_NOFOLLOW → 0 if undefined) is acceptable given Windows NTFS symlink privs.

### Parent directory mode enforcement

Both files check `parent.st_mode & 0o077 != 0` before reading, rejecting any group/world accessibility. This is the blocking layer ADR 0015 §1 locked — `~/.config/ergodix/` at 0o755 (the default) would undermine the file's 0o600 invariant. Well-documented with a loud remediation message.

### Atomic writes via temp + rename

`oauth.py::save_oauth_tokens()` writes to a `.tmp` file at mode 0o600, then renames. This closes the window between creation and chmod — no intermediate state where the file is world-readable. Belt-and-suspenders `os.chmod()` afterward is defensive and fine.

### Scope policy is tight

Locked to `drive.readonly` + `documents.readonly` per ADR 0001. The decision to deny broader scopes (e.g., `drive.file` for write, any `calendar` / `gmail` scope) without an explicit ADR is load-bearing and well-documented.

### Findings

1. **Medium**: Refresh token expiry / revocation detection flow is implicit.

In `oauth.py::load_or_acquire_credentials()`, when `creds.refresh()` fails with `RefreshError`, you clear the stale file and fall through to `acquire_oauth_credentials()`. This is the right recovery, but the error surface is opaque:

```python
try:
    creds.refresh(Request())
except RefreshError:
    clear_oauth_tokens()
    # Falls through — acquire_oauth_credentials() will run, user gets re-prompted
else:
    save_oauth_tokens(...)
    return creds
```

**Flag**: `RefreshError` is broad. Could be:
- Refresh token revoked (user action or Google timeout) — needs re-auth
- Network unreachable — might be transient, user might want retry
- Invalid token_uri, client_id mismatch — indicates a config problem

Today all three paths clear the file and re-auth. This works but doesn't distinguish. For migrate (the immediate caller), silent re-prompt on network flakiness could feel like "it hung and restarted from scratch" rather than "network hiccup, retrying."

**Recommendation**: Add error discrimination. Log / emit the raw `RefreshError` message so migrate can surface it:
```python
except RefreshError as e:
    if "revoked" in str(e).lower():
        # User revoked or Google timeout — clear and re-auth
        clear_oauth_tokens()
    else:
        # Network or config issue — raise for migrate to handle
        raise RuntimeError(f"OAuth refresh failed: {e}") from e
```

---

2. **Low**: No explicit validation of token_uri / client_id consistency on load.

When you deserialize from the token file, you call `_credentials_from_dict()` which reconstructs a `Credentials` object with fields like `token_uri`, `client_id`, `client_secret` from the saved dict. But there's no check that these match the current config:

```python
# oauth.py::load_or_acquire_credentials()
tokens = load_oauth_tokens()
if tokens is not None:
    creds = _credentials_from_dict(tokens)
    # creds.client_id == what was saved. But what if the user
    # rotated their OAuth credentials? Or someone committed the
    # wrong client ID by mistake?
```

**Flag**: If the OAuth client ID changes (e.g., moved to a different GCP project, or the user ran `auth set-key google_oauth_client_id` with a different value), the refresh token from the old client will fail but *silently* — the code path calls `creds.refresh()`, which fails with `RefreshError`, clears the file, and re-prompts. The user gets re-authenticated but doesn't see a loud "your client ID changed" message.

**Recommendation**: Before calling `refresh()`, validate that the client_id in the token file matches the current config:
```python
current_client_id = get_credential("google_oauth_client_id")
if creds.client_id != current_client_id:
    print(f"OAuth client ID changed (was {creds.client_id}, now {current_client_id}).")
    print("Clearing stale tokens and re-authorizing.")
    clear_oauth_tokens()
    # Fall through to acquire
```

3. **Low**: Refresh token lifetime and long-lived credentials.

Google's refresh tokens don't expire *per se*, but they **do invalidate** after:
- 6 months of non-use
- User revocation
- Password change
- Security events (Google-detected compromise)

The code handles revocation well (catch RefreshError, clear, re-auth). But there's no visibility into "this token is N days old and may be approaching the 6-month expiry." For a long-running migrate operation (or future continuous poller per ADR 0004), a 6-month-old token that goes stale mid-operation is silent failure.

**Flag**: If migrate is mid-walk and the refresh token becomes invalid, the next API call fails. The walk halts. On re-run, `load_or_acquire_credentials()` re-prompts. But there's no proactive "this token is old, let's refresh it before the walk starts" step.

**Recommendation** (deferred, not blocking): Consider adding a "token age" check on startup. If a token was last refreshed >90 days ago, proactively refresh (or re-auth if the refresh fails) before the walk begins. This keeps migrate from silently failing mid-operation.

---

4. **Low**: No rate-limit handling in the OAuth flow.

`load_or_acquire_credentials()` calls `flow.fetch_token(code=code)` and will raise `OAuthError` if the code is bad / expired / already-used. But Google also has rate limits on the token endpoint. If a user fat-fingers the code multiple times, they could hit rate-limit and see:

```
google.auth.exceptions.OAuthError: invalid_grant
(or some transient error from Google's rate-limiter)
```

**Flag**: The error message isn't obvious to the user. "Invalid code" and "you're rate-limited" look the same.

**Recommendation**: Catch `OAuthError`, check the response, and emit a friendlier message. E.g., if the HTTP status is 429 (too many requests), say "Google rate-limited us; wait a minute and try again." If it's an `invalid_grant`, say "Code didn't work. Make sure you pasted the full code, and that you're using a fresh one (codes expire after ~10 minutes)."

---

5. **Low**: Parent directory creation in save_oauth_tokens().

```python
parent.mkdir(parents=True, exist_ok=True)
```

This creates the parent directory but **does not set mode**. If `~/.config/` doesn't exist, `mkdir` creates it at the system umask, which is typically `0o022` (resulting in `0o755` after applying `~umask`). Then the subsequent `~/.config/ergodix/` is created inside a `0o755` parent.

But wait — C5 (the cantilever prereq) is supposed to ensure `~/.config/ergodix/` exists at mode `0o700`. So by the time `save_oauth_tokens()` runs, C5 should have already enforced parent mode.

**Flag**: If C5 runs first, this is fine. But if a user calls `save_oauth_tokens()` *before* C5 (e.g., during a direct OAuth test or a hypothetical standalone auth tool), the parent could end up at `0o755`. The code then correctly refuses to read from a loose parent later, but you end up in an inconsistent state.

**Recommendation**: Either (a) add mode enforcement to `save_oauth_tokens()` as a safety net:
```python
parent.mkdir(parents=True, exist_ok=True)
os.chmod(parent, 0o700)  # Enforce even if C5 hasn't run yet
```

Or (b) document loudly that C5 must run before any OAuth code that calls `save_oauth_tokens()`. Given that migrate is post-bootstrap (C5 is part of bootstrap), (b) is probably sufficient; but (a) is cheap insurance.

---

6. **Low**: Token file path resolution via local_config.py has no error recovery.

`oauth.py::_token_file_path()` tries to import `local_config.py` and read `TOKEN_FILE` from it. If the import fails (syntax error, broken dependency), it catches the exception and falls back to `.ergodix_tokens.json`. But the fallback is silent — no warning that the config was broken.

```python
with contextlib.suppress(Exception):
    # If anything goes wrong, fall back silently
    module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(module)
    token_file = getattr(module, "TOKEN_FILE", None)
    if isinstance(token_file, Path):
        return token_file
```

**Flag**: A broken `local_config.py` (which might have been edited by hand) will cause token lookup to silently use the wrong path. If the user configured `TOKEN_FILE = Path.home() / ".config" / "my-custom-token-store"` but made a syntax error, they'll end up with tokens in `.ergodix_tokens.json` and never know.

**Recommendation**: Log a warning if the import fails (so `--verbose` mode surfaces it), or raise an error during the auth flow with a "your local_config.py is broken" message. The silent fallback is too forgiving.

---

7. **Low**: Credentials dict serialization includes all fields, but deserialization is lenient.

`oauth.py::_credentials_to_dict()` saves `token`, `refresh_token`, `token_uri`, `client_id`, `client_secret`, `scopes`, `expiry`. But `_credentials_from_dict()` reads all of these and passes them to `Credentials(...)`. If a field is missing or `None`, the Credentials object might be created in an invalid state.

```python
def _credentials_from_dict(data: dict[str, Any]) -> Credentials:
    # ...
    return Credentials(
        token=data.get("token"),           # Could be None
        refresh_token=data.get("refresh_token"),  # Critical if token is expired
        # ...
    )
```

**Flag**: If a token file is corrupted (missing `refresh_token` or `token_uri`), the Credentials object will be created, but later calls like `creds.refresh(Request())` will fail cryptically (e.g., "NoneType has no attribute..." instead of "token file is corrupted").

**Recommendation**: Add explicit validation in `_credentials_from_dict()`:
```python
def _credentials_from_dict(data: dict[str, Any]) -> Credentials:
    required = ["token_uri", "client_id", "client_secret"]
    for key in required:
        if not data.get(key):
            raise ValueError(f"Token file missing or empty: {key}")
    # ... rest of reconstruction
```

### Review summary

**Ready to merge.** The security plumbing is solid:
- ✅ Symlink + TOCTOU protected
- ✅ Parent dir mode enforced
- ✅ Atomic writes
- ✅ Scope policy locked

**Pre-migrate checklist** (before `ergodix migrate --from gdocs` goes live):
1. Add token-age check (optional, can be v0.3.x)
2. Distinguish RefreshError causes (improves migrate UX)
3. Validate token client_id matches config (catches credential rotation mistakes)
4. Safe parent-dir mode enforcement in `save_oauth_tokens()` (cheap belt-and-suspenders)

None of these block the current code. They're quality-of-life for the migrate walker to come.
