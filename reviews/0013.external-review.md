# PR Review - ADR 0013

## Spike 0013 Amendment (ergodites naming + integrity section)

Scope reviewed:
- `spikes/0013-style-sentinel-and-certificate.md`
- `CHANGELOG.md`
- `VERSION`

### Findings

1. **Medium**: Integrity language risks over-stating security before signatures are in place.

In `spikes/0013-style-sentinel-and-certificate.md`, the new C.2 section correctly calls out the trust-anchor weakness of a hash-only manifest, but the current framing can still be interpreted as secure enough for adversarial environments. A salted hash dictionary by itself is tamper-evident only when the distribution channel is trusted; it does not resist a local attacker who can modify both ergodites and manifest.

**Recommendation:** Add one explicit sentence that phase 1 (hash-manifest only) is non-adversarial / trusted-distribution hardening, and that adversarial tamper resistance begins at signature verification (Approach 2).

2. **Low**: Open-question numbering became inconsistent after expanding ADR-X1.

`ADR-X1` now spans items 1-8, but `ADR-X2` restarts at 5 and `ADR-X3` continues at 10. The content is fine, but cross-referencing these items in future ADRs or discussions gets ambiguous.

**Recommendation:** Renumber for monotonic continuity (9-13 for ADR-X2, 14-18 for ADR-X3, etc.) or restart every section at 1 consistently.

### Consistency checks

- Version/changelog bump is coherent for this docs-only PR:
  - `VERSION` is `1.48.1`
  - `CHANGELOG.md` includes `[1.48.1] - 2026-05-10`
  - Unreleased note correctly points next docs PR to `1.48.2`

### Review summary

No blocking issues found. This PR is reviewable as-is; the two findings are documentation clarity improvements rather than regressions.
