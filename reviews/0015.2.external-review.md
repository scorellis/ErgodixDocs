## PR #74 — Migrate CLI wiring (`ergodix/cli.py`)

### Strengths

**Author resolution is well-layered.** `_read_author_from_local_config()` → `_git_config_user_name()` → empty string is exactly the right precedence chain per ADR 0015 §3. Both fallbacks are silent (no crash on broken config, no crash on missing git). The git subprocess call has a 5-second timeout and `check=False` — can't hang the CLI.

**`--check` / `--force` mutex handled at the CLI layer, not in migrate_run.** That's the right place — the UI owns the validation; the orchestrator doesn't need to care.

**`docs_service` is only acquired for `--from gdocs`.** The lazy import means a `.docx` migration doesn't trigger OAuth prompts. Clean separation.

**Error messaging is informative.** Missing corpus path gives a clear "set CORPUS_FOLDER in local_config.py or pass --corpus" message before sys.exit(1). Exit codes are meaningful (0 = success, 1 = any failures, 2 = bad args).

### Findings

**Low: `_read_author_from_local_config()` silently swallows all exceptions.**

```python
try:
    module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(module)
except Exception:
    return None
```

Unlike `_token_file_path()` (which now emits a `UserWarning` after PR #72's finding #6 fix), this function discards broken-config errors silently. If the user's `local_config.py` has a syntax error, migrate proceeds with an empty author field, and the user has no indication their config was ignored.

**Recommendation:** Mirror the pattern from `oauth.py::_token_file_path()` — emit a `UserWarning` on exception before returning `None`. The message text can be brief: `"local_config.py could not be loaded; AUTHOR field ignored."` This keeps the function non-fatal but visible.

**Low: `counts_str` formatter is fragile on non-standard count keys.**

```python
counts_str = ", ".join(f"{k}={v}" for k, v in sorted(result.counts.items())) or "(no files)"
```

`sorted()` on arbitrary string keys produces alphabetical output which may differ from logical importance order (e.g. `failed=2` at the end after `imported=10, skipped=3`). Not a bug, but the UX could put `failed` first when non-zero.

**Recommendation (optional):** Emit failed count first when non-zero: `f"failed={result.counts['failed']}, ..."`. Low priority — alphabetical is readable enough.

## PR #75 — `.docx` importer (`ergodix/importers/docx.py`)

### Strengths

**Module structure mirrors gdocs importer exactly.** `NAME`, `EXTENSIONS`, `extract()`, private helpers. The registry contract is clean.

**`**_kwargs` absorbs orchestrator passthrough.** Docx doesn't need `docs_service`; the swallow pattern keeps `migrate_run`'s `importer.extract(path, **kwargs)` call shape working without per-importer dispatch.

**Lazy import of `docx`.** `from docx import Document` inside `extract()` means test collection doesn't fail on machines without `python-docx`. Consistent with the pattern in gdocs importer.

**Style-name matching is tolerant.** `_HEADING_STYLES` maps named styles; `List Bullet` check via `startswith("List Bullet")` covers "List Bullet 2", "List Bullet 3", etc. without enumerating them.

**Bold/italic run-level rendering is correct.** `***text***` for bold+italic, `**text**` for bold, `*text*` for italic. Order of checks (bold+italic first) avoids the double-wrap bug.

### Findings

**Low: `_render_paragraph` accesses `paragraph.style.name` with a two-step null guard.**

```python
style_name = ""
if paragraph.style is not None and paragraph.style.name:
    style_name = paragraph.style.name
```

`python-docx` always sets `paragraph.style` to at least the "Normal" default when loading a standard `.docx`; a `None` style is only possible for pathological/malformed documents. The guard is correct and safe — just noting it's defensive.

**Low: no explicit test for `_render_paragraph` with a `None` style (malformed doc path).**

The guard exists, but there's no test that exercises a `None`-style paragraph. Worth adding for completeness if the importer is expected to be resilient against corrupt Scrivener exports.

**Low: image extraction is deferred to chunk 6 but the module-level docstring was updated in chunk 6 without a stub test.** No impact on functionality; the skipped-silently behavior for images in v1 is correct. Just noting the chunk boundary is clean.

## PR #73 — Spike 0014 (`spikes/0014-form-analyzer-ergodite.md`)

*Documentation-only PR. TDD rules don't apply. Commentary is design-focused.*

### Strengths

**The three sub-checks are orthogonal and well-scoped.** Readability (settled science, public formulas), rhetorical eloquence (pattern-based, LLM optional), and Fibonacci structure (author-specific, correctly parked behind a precondition) have different dependency profiles and can fail independently. Bundling them under one ergodite but making each toggleable via settings is the right call.

**`textstat` reference is correct.** The library bundles all 7 readability formulas and is pure-Python. Worth checking whether it's being actively maintained before adding it as a runtime dep, but it's the standard tool for this problem.

**The Devil's Toolbox dependency for rhetorical eloquence is flagged as a precondition.** The spike doesn't pretend it can ship rhetoric detection without the toolbox reference. Cross-reference to Spike 0013 is explicit.

**Fibonacci arc is correctly gated.** "Get the Fibonacci writing prompt into the repo first" is the right precondition — the spike can't spec the check without the author's explicit arc shape.

**The single-ergodite vs. three-ergodite tradeoff table is clear.** The lean decision (single ergodite, per-sub-check toggles) is well-argued.

### Findings

**Medium: the `ImageFetcher` alias and layered fallback in gdocs importer (chunk 6b) may benefit from a call-out here.** The spike discusses the gdocs image-fetching mechanism (contentUri + Drive-authenticated HTTP) at a high level but doesn't flag that the retry / error-handling surface for image fetch failures is currently "log-and-skip." If the author has 50-image-heavy chapters, a single batch failure silently drops all images rather than re-queueing. This is a parking-lot concern, but worth flagging in the spike or in a CHANGELOG note.

**Low: Section D's settings example uses a dotted-key syntax** (`form_analyzer.grade_level: on`) but the settings cascade (ADR 0014 via `settings/floaters/<name>.toml`) presumably flattens TOML keys differently. Worth verifying the settings key naming against the TOML schema before ADR-X1 lands.

## PR #76 — Hermetic e2e tests (`tests/test_migrate_fixture.py`, `examples/migrate-fixture/`)

### Strengths

**The fixture-shape test is the right first test.** `test_fixture_shape_on_disk` pins the contract — if a refactor accidentally moves fixture files, it surfaces as a fast unit failure before any e2e test runs. Good defensive pattern.

**The mock Docs service is layered cleanly.** `_make_mock_docs_service()` uses side_effect to dispatch on `documentId` — tests get deterministic, network-free results without touching real Docs API. Fail-loud on unexpected doc IDs (`RuntimeError(f"unexpected doc {documentId}")`) is correct — it prevents a silently-empty extract from masking a bad fixture setup.

**`_copy_fixture` → tmp_path prevents corpus mutation.** Archive moves during `migrate_run` would permanently alter the checked-in fixture without this. The copy is cheap.

**`_CHAPTER_1_DOC` shape is spec-faithful.** The canned Docs API JSON uses the real API shapes (`paragraph.elements`, `textRun`, `paragraphStyle.namedStyleType`), not invented shortcuts. Tests using this will catch regressions in the gdocs renderer.

### Findings

**Low: `Notes.gsheet` is in the fixture but there's no test asserting it's skipped.** The walker should skip `.gsheet` files (not a recognized importer extension). The fixture's `test_fixture_shape_on_disk` asserts the file exists, but I don't see an assertion that a full migrate run produces zero errors for the `.gsheet` file (i.e. it's classified as "skipped" not "failed"). Worth adding to an e2e test to pin the "unknown extension → skip" behavior.

**Low: The `.docx` binary is checked in as `Book 1/Chapter 2.docx`.** This is deliberate (hermetic), and there's a builder script (`tests/build_migrate_docx_fixture.py`). Ensure the script is kept in sync if the `.docx` fixture is ever regenerated — a stale binary with a current builder script will produce silent diffs in test output.

## PR #77 — OAuth rate-limit messaging (`ergodix/oauth.py`)

### Strengths

**Heuristic classification is honest about being heuristic.** The docstring says "heuristic-only" and explains why (Google wraps errors opaquely). Three buckets (invalid_grant, rate-limit, generic) cover the common cases without overpromising precision.

**Re-raises the original exception.** `_emit_token_exchange_diagnostic` is called before `raise`, so the caller gets the original exception with its type and traceback intact. A migrate caller can still catch `OAuthError` specifically if needed. Correct pattern.

**Token keywords cover Google's historical variants.** `"invalid_grant"`, `"expired"`, `"redeemed"`, `"used"` for the bad-code bucket; `"429"`, `"rate"`, `"quota"`, `"too many"` for the rate-limit bucket. The string-in-lower match is tolerant of API response evolution.

### Findings

**Low: "redeemed" and "used" as heuristic keywords.**

These are strings occasionally found in OAuth error messages but they're also common English words that might appear in unrelated error contexts. A malformed `token_uri` could theoretically produce a DNS error message that contains "used" (e.g. "address already used"). The probability is low but the mis-classification would give a misleading "bad code" message.

**Recommendation:** Consider anchoring to `"code has been"` or `"already used"` (phrase-level) rather than bare `"used"` for this bucket. Not urgent — the existing heuristic covers 99% of real cases.

**Low: No tests for `_emit_token_exchange_diagnostic` directly.** The rate-limit and generic branches are tested indirectly (through `acquire_oauth_credentials` mock tests), but a direct test of each heuristic bucket would be more explicit and easier to maintain if the keyword list changes.

## Unmerged: `feature/migrate-chunk-6-recover` (docx image extraction)

### Strengths

**`_extract_images` walks `document.part.rels.values()` using the `"image"` substring match.** This is more robust than matching the full Office Open XML namespace URI, which varies across Word versions. The `_IMAGE_RELTYPE_TOKEN` constant with its `# noqa: S105` annotation is honest about what it is.

**Sequential `img-NNN<ext>` naming is idempotent-safe.** Re-running chunk 6 over the same chapter would produce `img-001.png` again (if media_dir already has it, `write_bytes` overwrites). This is acceptable for v1 — images are deterministic from the source `.docx` — but worth noting for re-run semantics.

**`_suffix_for_image_part` falls back to `.bin`.** A malformed docx with no file extension in its `partname` gets a safe, non-crashy extension. The fallback is correct.

**`None` on `--check` prevents filesystem side effects in dry-run.** The `media_dir = None if check else ...` in `migrate_run` correctly suppresses image extraction during dry-run. Clean.

### Findings

**Medium: `media_dir.mkdir(parents=True, exist_ok=True)` inside `_extract_images` has no mode enforcement.**

```python
media_dir.mkdir(parents=True, exist_ok=True)
```

Contrast with `save_oauth_tokens()` which sets `mode=0o700` on `mkdir`. The `_media/` directories are not security-sensitive (they contain images, not credentials), but this inconsistency means they're created at the umask default (typically `0o755`). That's probably fine for image media, but it's worth an explicit decision rather than an implicit default.

**Recommendation:** Document in a comment that `_media/` dirs are intentionally world-readable (they hold chapter images, not secrets). This makes the inconsistency with auth dirs intentional rather than an oversight.

**Low: no test for the `.bin` fallback in `_suffix_for_image_part`.**

The `.bin` path requires a `part` with no `partname` attribute (or one with no extension). Worth adding a targeted test if the function is expected to be resilient against malformed docx files.

## Unmerged: `feature/migrate-chunk-6b-gdocs-images` (gdocs image extraction)

### Strengths

**`ImageFetcher` type alias is well-designed.** `Callable[[str], bytes | None]` gives tests a clean injection surface without needing to mock `AuthorizedSession`. The lazy build (`_build_default_image_fetcher()` only on first image encountered) avoids OAuth prompts on image-free documents.

**`_guess_image_extension` uses magic bytes first, URI suffix second.** Magic bytes are more reliable than content-type headers (which can be wrong) or URI extensions (which can be absent). The WebP check (`b"RIFF"` + `b"WEBP"`) is correct.

**"Images that fail to fetch → silently dropped" is the right call for v1.** Body content renders; the user gets a manifest with the files that worked. The alternative (failing the whole chapter on one bad image) is worse. The docstring is explicit about this behavior.

**Inline image traversal is defensive.** Every `element.get(...)` call falls through on `None` or wrong type. The body walk can't crash on malformed Docs API JSON.

### Findings

**Medium: `_build_default_image_fetcher` is referenced in the docstring and code but not shown in the diff reviewed.** The implementation is present (the diff only showed part of the file), but it's worth verifying it:
1. Builds an `AuthorizedSession` from `load_or_acquire_credentials()`
2. Handles HTTP errors (non-200 responses) by returning `None` rather than raising

If it raises on a 403 or 404, a missing image would propagate as an exception rather than a silent skip, contradicting the "failed fetches are silently dropped" contract.

**Recommendation:** Verify `_build_default_image_fetcher` returns `None` on all non-200 HTTP responses. If not, wrap the HTTP call in `try/except` to enforce the contract.

**Low: `ext = _guess_image_extension(image_bytes, content_uri)` — the `content_uri` argument is accepted but the current diff only shows magic-byte sniffing.** If magic bytes don't match (unusual format), does the URI fallback handle Docs' content URLs (which look like `https://lh3.googleusercontent.com/...`)? Those typically don't have a meaningful file extension in the path segment.

**Recommendation:** If the URI fallback is implemented, test it against a Docs-style `lh3.googleusercontent.com` URL (which would produce `.bin` as the fallback, which is acceptable but worth pinning in a test).

## Unmerged: `feature/oauth-backlog-3-and-7` (OAuth findings #3 + #7)

### Strengths

**Finding #7: Required-field validation in `_credentials_from_dict` is the right approach.** The three-field check (`token_uri`, `client_id`, `client_secret`) covers the fields that cause cryptic NoneType errors on refresh. Collecting all missing fields before raising (rather than raising on the first one) gives the user the full picture in one pass.

**The error message is actionable.** "The token file may be corrupted or pre-1.0; clear it and run migrate again to re-authenticate." tells the user exactly what to do.

**Finding #3: `refresh_token_issued_at` is stamped at acquire-time, not at access-token-refresh-time.** This is correct — the access token rolls but the refresh token doesn't. The code preserves the original `refresh_token_issued_at` across access-token refreshes (`if "refresh_token_issued_at" in tokens: refreshed["refresh_token_issued_at"] = tokens["refresh_token_issued_at"]`). Well-reasoned.

**Inform-don't-block pattern for staleness warning.** 90-day threshold as a "heads-up" before the 6-month Google invalidation window is sensible. The user gets lead time without being forced to re-auth unnecessarily.

**Graceful no-op for older token files.** `_warn_if_refresh_token_stale` returns immediately if `refresh_token_issued_at` is missing — backward compatible with token files written before this chunk.

### Findings

**Low: `refresh_token_issued_at` format uses `strftime("%Y-%m-%dT%H:%M:%SZ")` but `_warn_if_refresh_token_stale` uses `datetime.fromisoformat()`.**

`fromisoformat()` in Python 3.11+ handles the `Z` suffix (prior to 3.11 it didn't). Since the project requires Python ≥3.11 (CLAUDE.md), this is fine. But worth a comment noting the Python version dependency, since the code is `from __future__ import annotations`-first and implies portability awareness.

**Low: The 90-day threshold is a constant buried in `_warn_if_refresh_token_stale` but is not parameterisable.** If Google's policy changes, or the team wants a different lead time, it requires editing the function. A module-level constant (`_REFRESH_TOKEN_STALE_DAYS = 90`) would make this more visible.

**Recommendation:** Extract the threshold as a module constant with a comment referencing Google's 6-month policy. One line of change, improves maintainability.

**Low: No test for finding #7 with a partially-corrupted token file** (i.e. some required fields present, one missing). Tests should cover:
- All three fields missing
- One specific field missing (e.g. only `client_secret` absent)
- All three present (happy path, no exception)

Partial-field corruption is the realistic failure mode when a user hand-edits the token file.
